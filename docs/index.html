Evaluation Dimensions

SDS evaluation is best treated as a layered construct: component scores reveal local defects, interaction scores reveal dialog behavior, and user studies reveal whether the system actually works for people.

Research Map

1. System Components

Classic SDS work decomposes interaction into ASR, language understanding, dialog state, policy, generation, and speech output. Modern spoken LLM systems compress that pipeline, but the old component boundaries remain useful diagnostic axes.

2. Human Evaluation

Human studies measure perceived quality: task success, satisfaction, cognitive load, naturalness, trust, and recovery from errors. SASSI, ITU-T P.851, Wizard-of-Oz studies, pairwise preference tests, and fine-grained rubrics cover different reliability and cost tradeoffs.

3. Automatic Evaluation

Automatic metrics include WER, semantic error rate, dialog-state accuracy, slot F1, inform/success, policy reward, turn count, latency, BLEU-like response scores, reference-free learned metrics, and LLM-as-judge protocols. Their validity depends on correlation with human outcomes.

4. Benchmark Risk

Shared datasets made progress measurable, but they can overfit the field to annotation artifacts, text-only assumptions, or narrow domains. MultiWOZ correction papers and spoken-model surveys show why benchmark governance is part of evaluation.

Extend This Site

Add new entries in docs/data/literature.js. Each record has year, component, type, evaluation, title, authors, venue, url, and note. The interface updates automatically without a build step. A readable sorted companion bibliography is available at literature.md.

For GitHub Pages, publish the repository from the /docs folder in repository settings.

Source-backed starting points include ACL Anthology, ITU-T P.851, W3C VoiceXML, NIST Communicator analysis, Springer/PMC evaluation surveys, TACL, arXiv, and publisher DOI landing pages. This is a curated atlas rather than an exhaustive systematic-review database.
